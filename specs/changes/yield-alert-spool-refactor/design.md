# Design: yield-alert-spool-refactor

## Summary
Today the yield-alert-center page runs a split data flow: the alert list is served from the `yield_alert_dataset` DuckDB spool (built once from `ERP_WIP_MOVETXN_DETAIL` via `execute_primary_query`), while the trend and Summary-card views run *live* Oracle queries against the `ERP_WIP_MOVETXN` aggregate table at request time (`query_yield_trend` / `query_yield_summary` in `yield_alert_service.py`), and reject linkage is a *second* Oracle round-trip (`_compute_reject_linkage`) issued after the spool write. This change collapses everything onto a single source-of-truth: one initial `ERP_WIP_MOVETXN_DETAIL` spool pull feeds all four views (trend, Summary cards, heatmap, alert list), computed in DuckDB at view-serve time. The live Oracle query path for trend/summary is retired; reject linkage is folded into the primary pull; a `process_type` selector (GA% packaging / GC% point-test, scoped by `WIP_ENTITY_NAME LIKE`) is added at the primary query stage and becomes part of the spool query-id; the alert list gains a `SOURCE_CODE` (LOT) dimension; and the now-redundant `PACKAGE IS NOT NULL` filter is dropped for GA%. The module boundary moves: `yield_alert_service.py` ceases to be a request-time Oracle gateway for trend/summary and the spool becomes the only post-build read surface.

## Key Decisions

**D1 — Reuse `yield_alert_dataset` namespace with a schema-version bump.** The existing namespace is reused rather than minting a new one, because the spool *contents* change (new columns, new source table, new scope dimension) but the spool *role* (single per-date-range dataset for all four views) is unchanged. The established pattern (cache-spool-patterns §`_SCHEMA_VERSION`) is to embed a module-level schema-version integer in the query-id hash so a redeploy that bumps it orphans every stale parquet by key-miss and readers rewrite automatically. We bump `_CACHE_SCHEMA_VERSION` (4 → 5) and additionally fold `process_type` into the query-id, so GA% and GC% datasets coexist as distinct keys under the one namespace. A new namespace would force changes to `_ALLOWED_NAMESPACES`, its test parametrization, warmup registration, and the frontend spool path for zero benefit.

**D2 — No live-query fallback: expired spool returns 410, client re-triggers the query.** Once trend/summary stop querying Oracle live, there is no request-time Oracle path left to fall back to when a spool is missing or expired. The existing spool-serve contract already returns `410 Gone` when a parquet has expired or never existed (`spool_routes.py`), and the frontend already knows how to re-trigger the primary query on that signal. We keep that single degradation mode rather than re-introducing a live Oracle path solely as a trend/summary fallback. This avoids maintaining two divergent code paths (live SQL vs DuckDB) that can silently disagree — the exact dual-source inconsistency this refactor exists to eliminate.

**D3 — Reject linkage computed via a single join in the primary Oracle pull.** `REJECT_LINKED` becomes a boolean column produced by a join inside `_PRIMARY_DETAIL_SQL`, replacing the separate `_compute_reject_linkage` round-trip that ran after the spool write (`execute_linkage_query`). Because linkage is now materialized into the spool parquet at build time, every view reads it for free with no second Oracle hit and no `linkage_ready=false` interim state. This removes the asynchronous linkage-enrichment step (`_enrich_alerts_with_linkage`) from the request path.

**D4 — `SOURCE_CODE` added to alert GROUP BY/SELECT; alert totals are provably unaffected.** Adding the LOT dimension increases spool row count ~2.4x (417,928 → 1,001,283 rows/month for GA%) because scrap-only LOT rows fan out the grouping. The confirmed data invariant (verified by direct Oracle query, recorded in business-rules YA-04/YA-06): **`SOURCE_CODE NOT NULL` rows are 100% scrap-only with `TRANSACTION_QTY = 0`.** Therefore the added rows contribute to `SCRAP_QTY` and to the LOT display dimension but contribute zero to the `TRANSACTION_QTY` denominator — yield totals and work-order granularity are unchanged. This invariant must be pinned by a unit test on the spool row shape.

**D5 — Invalid `process_type` returns VALIDATION_ERROR, not a silent default.** GA% is the *default when the param is absent*, but an explicitly-supplied invalid value (anything other than the GA%/GC% enum) returns `VALIDATION_ERROR` rather than being coerced to GA%. Silent coercion would let a client believe it queried point-test data while actually receiving packaging data — a correctness hazard for an engineer comparing yields across process types. Absent param → documented GA% default; present-but-invalid → reject loudly.

## Rejected Alternatives

**Alt-A — Keep the live Oracle trend/summary path as a fallback.** Rejected. Retaining `ERP_WIP_MOVETXN`-based live queries alongside the spool re-creates the dual-source divergence (different table, different filter semantics for PACKAGE/LINE/TYPE on numerator vs denominator) that motivated this change. Two code paths that must produce identical numbers but cannot be guaranteed to is a permanent correctness liability; the 410 → re-query mode (D2) is simpler and self-healing.

**Alt-B — Separate spool namespaces for GA% and GC%.** Rejected. The two process types share one source table, one column set, one builder, and one set of views; they differ only by a `WIP_ENTITY_NAME LIKE` predicate. Distinguishing them by including `process_type` in the query-id (D1) gives full key isolation without duplicating namespace registration, whitelist entries, warmup wiring, and frontend spool paths.

**Alt-C — Keep `_compute_reject_linkage` as a separate Oracle call.** Rejected. The separate round-trip adds request-time latency, an extra Oracle connection, and a transient `linkage_ready=false` state the frontend must poll past. Folding linkage into the primary pull (D3) makes it a static spool column — one Oracle query instead of two, and linkage is ready the instant the spool is.

## Migration / Rollback

**Forward deploy (no manual step):** Bump `_CACHE_SCHEMA_VERSION` (and add `process_type` to the query-id hash) in the *same commit* as the SQL/column change. On redeploy, every existing `yield_alert_dataset` parquet key-misses against the new schema version and is silently re-built on next query; stale parquets expire on their normal TTL. No manual `rm` is required for the forward path.

**Rollback runbook:** Revert the code commit, then purge the new-schema parquets that the reverted code can no longer read: `rm -f tmp/query_spool/yield_alert_dataset/*.parquet`. Because the schema-version integer is part of the query-id, the reverted (older) code will also key-miss the new parquets and rebuild; the `rm` is a belt-and-suspenders cleanup to reclaim disk immediately and avoid orphaned files accumulating. The parquet-rm step must be listed in the rollback section of the deploy doc and ci-gates per the cache-spool pattern. No env-var or DB migration is involved — rollback is code-revert + parquet purge only.

## Open Risks
- 2.4x spool volume (~1M rows/month for GA%) must be validated under the stress/soak lanes for both build time and DuckDB browser-query latency; the `rq_yield_alert_worker` warmup interacts with the larger dataset (soak lane).
- `REJECT_LINKED` join correctness depends on the same workorder-normalization the old `_compute_reject_linkage` applied (upper/trim); the join predicate must reproduce that normalization or linkage counts will silently drift.
- GC% (wafer-sort) retains `PACKAGE=NA` lots; the PACKAGE-filter removal is verified safe *only for GA%* — GC% handling of `PACKAGE=NA` must be covered by a data-boundary test.
