# Design: nav-config-to-code

## Summary

Option B relocates the navigation **structure** source-of-truth (drawer id/name/order/admin_only, page→drawer assignment, page order, page display name) out of the runtime CMS (`data/page_status.json` + admin endpoints) into a code-side manifest in the frontend, collapsing the 3-way duplication (`nativeModuleRegistry.js` mount gate / `routeContracts.js` policy meta / `page_status.json` placement). The only thing that stays runtime-writable is a per-page binary `status` (`released` vs `dev`). The crux is **where structure+status re-merge**: today the merge happens server-side in `app.py::portal_navigation_config()` (the real AC-1 surface, NOT `GET /api/drawers`) and is re-normalized client-side in `navigationState.js`. After this change, structure comes from the manifest (client) and status from a slim runtime route→status map; the merge moves fully client-side. Blast radius is admin-only except the rendered menu, which must stay structurally identical (AC-1).

## Key Decisions

- **New `navigationManifest.js` module, structure-only — NOT extending `routeContracts.js`.** `routeContracts.js` owns per-route *policy* (renderMode/visibilityPolicy/scope/owner) and is keyed by single route; the manifest owns *menu placement* (drawer grouping + ordering + display name). Mixing them recreates the duplication we are removing and bloats a 398-line policy file. The manifest references routes that must already exist in `nativeModuleRegistry` (mount gate) and `routeContracts` (policy); it adds neither component nor CSS wiring. → Rejected "extend routeContracts.js": conflates two concerns, and the title/owner fields there are not the menu display-name/grouping. → Rejected a backend-owned manifest copy: would re-introduce a server source-of-truth and a sync burden; the only frontend consumers (portal-shell, admin-pages) can import the JS module directly.

- **Merge moves fully client-side; `/api/portal/navigation` is inverted to a status feed.** Structure lives only in the manifest (frontend), so the server can no longer assemble drawers. `portal_navigation_config()` keeps emitting `is_admin`/`admin_user`/`admin_links`/`features` (auth-derived, must stay server-authoritative) plus a `statuses` route→status map; it stops emitting `drawers`. `navigationState.js` merges manifest structure × statusMap, applying the existing `canViewPage`/`admin_only` rules unchanged. → Rejected "keep server-side merge, feed it the manifest": the server has no access to a frontend JS module without duplicating it. → Rejected a brand-new `/api/pages/status` endpoint: `/api/portal/navigation` is already the bootstrap call App.vue makes; reshaping it avoids a second round-trip and keeps the auth/admin_links payload co-located.

- **`GET /api/pages` slims to route+status; admin UI joins names from the manifest.** After the change the backend does not own display names or drawers, so it cannot return them. The admin status-toggle list needs (route, name, status); it obtains name/drawer by joining the manifest (imported directly in `admin-pages`) with the runtime status map from `GET /api/pages`. Backend's residual ownership = the **valid-route set** (for `PUT /api/pages` write validation) + the **status map**. → Rejected "backend keeps returning names": forces names to live in two places again.

- **Fail-safe default = visible (`released`).** A route absent from the status map, a missing file, or a legacy full-CMS file must never error and must default to visible — mirroring today's `_can_view_page_for_user` (status `None` → visible) and `canViewPage` (`|| 'released'`). The manifest may mark a route's `defaultStatus: 'dev'` (only `/admin/dashboard` today) so a freshly-shrunk/missing store still hides the one dev page. → Rejected "default to dev/hidden on missing data": would blank the entire menu on a read failure, violating AC-1 under degradation.

- **Drawer-id rename + explicit distinct orders (AC-5).** Internal ids only: `drawer-2`→`history-reports`, `drawer`→`query-tools`, `drawer-3`→`trace-tools`; keep `reports`/`dev-tools`/`eap-analysis`; **drop empty `test`**. Assign distinct integer drawer orders 1..6 (reports=1, history-reports=2, query-tools=3, trace-tools=4, dev-tools=5, eap-analysis=6) to resolve the `dev-tools`/`trace-tools` order-4 tie deterministically while preserving the current name-tiebreak visual order. Within trace-tools, give `/query-tool`=1, `/mid-section-defect`=2, `/material-trace`=3 to resolve the order-3 tie (preserving current `mid-section-defect` before `material-trace` name-tiebreak order). All display names + page memberships preserved verbatim; standalone routes (`/`, `/wip-detail`, `/hold-detail`) stay outside any drawer (already the code-owned `STANDALONE_DRILLDOWN_ROUTES` set, which also includes `/anomaly-overview`, registered but menu-absent). → Rejected "keep colliding orders + rely on name tiebreak": leaves rendered order ambiguous and fragile to future renames.

## Migration / Rollback

**Forward:** Author `navigationManifest.js` reproducing the current rendered layout 1:1 (drawer ids renamed, `test` dropped, distinct orders, names/memberships verbatim; `defaultStatus:'dev'` on `/admin/dashboard`). Shrink `data/page_status.json` to a route→status map — only `/admin/dashboard:'dev'` is non-default today; all others `released` (and may be omitted, since absence defaults to released). The `page_registry` read path treats the legacy full-CMS file as input: if the file has the old `{pages:[...],drawers:[...]}` shape, derive the route→status map from `pages[].status` and ignore the rest (no forced rewrite, no error) — so an un-migrated file still serves correct statuses. Endpoint/CRUD removal and the admin-UI trim ship together with the slimmed `GET /api/pages`/`/api/portal/navigation`. Regenerate `get_admin_pages.json`; retire `get_admin_drawers.json` + `delete_admin_drawers_id.json`; record the breaking removal in `contracts/CHANGELOG.md`.

**Rollback:** Revert the code commit — restoring the drawer endpoints, the full `set_page_status` signature, the structure-emitting `portal_navigation_config`, `navigationState`'s backend-drawer input, and the admin DrawerManagementPanel. An already-shrunk `page_status.json` is forward-and-backward safe: the restored `_migrate_navigation_schema` already rebuilds the `drawers` array from `DEFAULT_DRAWERS` + `LEGACY_NAV_ASSIGNMENTS` when `drawers` is absent and persists it on first read, so a route→status-only file self-heals to the legacy CMS shape on the first post-rollback load (note: it heals to DEFAULT_DRAWERS, not the renamed ids — operators wanting the exact pre-change menu restore the file from git). No data is destroyed by the shrink; the dropped CMS fields are reconstructable from code defaults. No DB migration, queue, or cache is involved, so rollback is a code revert plus (optionally) `git checkout` of `data/page_status.json`.

## Open Risks

- **CER-003 (blocking precision, not blocking the design):** `src/mes_dashboard/app.py` holds the real AC-1 surface (`portal_navigation_config`) and is outside Allowed Paths. spec-architect read L1105-1245/L260-310 to ground Decision 2/3; backend-engineer and contract-reviewer need it formally approved (handler edit + AC-1 payload contract). Until approved, the "invert `/api/portal/navigation` to a status feed" decision is provisional.
- **AC-1 regression is a JS-object diff, not a server diff.** Because the merge moves client-side, the menu-parity proof must compare the *rendered* `buildDynamicNavigationState` output (drawers→pages→order→name→visible-set) for admin and non-admin against the captured baseline in `current-behavior.md`. A manifest typo (wrong order/drawerId/name) silently breaks AC-1 with no backend signal — test-strategist must assert the built nav tree, not just endpoint shape.
- **`/api/portal/navigation` `diagnostics.contract_mismatch_routes` / shell-contract validation** currently runs server-side over the structure it assembles. Once structure leaves the server, that validation has nothing to validate; decide whether to drop it or move an equivalent check into the client manifest-vs-registry consistency test. (Lower risk: diagnostics only logs/annotates.)
- **`api_public` / `db_scan` keys in `page_status.json`** are read by `is_api_public()` and a tools script, unrelated to navigation. The shrink must preserve `api_public` semantics (or relocate it) — do not let "shrink to route→status map" silently drop the `api_public` gate. Flag for backend-engineer + contract-reviewer.
