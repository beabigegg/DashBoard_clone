# Design: production-achievement-kanban

## Summary
Adds a new filterable "生產達成率" report page under the existing 生產輔助 drawer. Three architectural additions drive the design.md requirement: (1) a NEW single-flag authorization primitive `can_edit_targets`, independent of and additive to the existing binary `admin_required`, backed by a new MySQL table and failing closed; (2) a deliberate deviation from the SQLite→MySQL `sync_worker` dual-layer pattern — the target-value and permission tables read/write MySQL directly via `core/mysql_client.get_mysql_connection()` because they need immediate read-your-writes consistency across multi-worker/multi-host, which the ~10-min one-way, no-read-back sync cannot provide; and (3) since the project has no MySQL migration runner, the two new tables are created by a committed idempotent DDL script applied manually by an operator before first deploy. The Oracle read path re-implements two Oracle functions (`PJ_GET_CLASSCODE_F`, `PJ_GET_OUTPUTDATE_F`) as SQL CASE expressions inside the source query.

## Key Decisions

- **New permission function lives in `core/permissions.py`, not a new module.** Rationale: it is a session-identity authorization check, the same conceptual layer as `admin_required`/`login_required`; co-locating keeps the trust boundary discoverable in one file → rejected a separate `core/target_permissions.py`: reason — fragments the auth surface for one boolean capability, over-engineering the "minimal scope" the request mandates. The MySQL row lookup is delegated to `production_achievement_permission_service` (permissions.py stays free of DB coupling; it calls the service and interprets the result).
- **`can_edit_targets()` signature and failure-closed semantics.** `can_edit_targets(user_identifier: str | None = None) -> bool` (defaults to `get_owner_token()`/session username) and decorator `targets_edit_required(f)` returning `forbidden_error` (403). It MUST return `False` (deny) when: no whitelist row exists, `MYSQL_OPS_ENABLED=false`, or any MySQL exception is raised — the service wraps the lookup in try/except and returns `False` on error (data-shape §3.27 default-deny). → rejected fail-open-on-error: reason — a new auth boundary that opens targets to everyone on a DB blip is the exact high-risk failure this review exists to prevent.
- **Direct MySQL, not sync_worker.** Target values and permissions are the read-your-writes denominator/gate of the feature; the established `sync_worker` is one-way SQLite→MySQL, batch (~600s), INSERT-IGNORE/REPLACE with no read-back and MySQL-offline tolerance — structurally unable to serve an immediate-consistency edit-then-see flow across workers. New services use `with get_mysql_connection() as conn` (already commits on success / rollback on exception). Writes use `INSERT ... ON DUPLICATE KEY UPDATE` keyed on the unique constraint (`(shift_code, workcenter_group)` for targets; `user_identifier` for permissions). → rejected extending sync_worker: reason — wrong consistency model and couples an eventual-consistency log pipeline to a synchronous auth path.
- **Per-endpoint MySQL-failure behavior.** Reads (`GET .../report`, `GET .../targets`) degrade: when `MYSQL_OPS_ENABLED=false` or MySQL is unreachable, `target_qty=null` → `achievement_rate=null` (PA-07), HTTP 200, never 500. Writes (`PUT .../targets`, `PUT .../permissions/{id}`, `GET .../permissions`) cannot degrade — return 503 `SERVICE_UNAVAILABLE` when OPS disabled/unreachable. Permission-gated write returns 403 `FORBIDDEN` when `can_edit_targets` is false. The Oracle report read is independent of MySQL health — a MySQL outage never blocks the achievement numerator, only nulls the denominator.
- **Shift_code/output_date as SQL CASE inside the Oracle query, not post-hoc Python.** Rationale: achievement-rate grouping is `GROUP BY (output_date, shift_code, workcenter_group)` and the PA-05 qualifying predicate MUST run at the Oracle source (filters `DW_MES_LOTWIPHISTORY` before aggregation, per the request's performance note). Since the grouping key columns must exist at the DB to let Oracle do the `SUM`/`GROUP BY`, computing shift_code and output_date as CASE expressions in the same SELECT keeps grouping server-side and avoids fetching every qualifying row into Python. → rejected pure-Python re-impl of `PJ_GET_CLASSCODE_F`/`PJ_GET_OUTPUTDATE_F`: reason — would force fetch-all-qualifying-rows then group in-process, defeating the source-side aggregation. A thin Python mirror of both functions is still authored for unit-test boundary assertions (PA-01..PA-04) but is not on the query hot path.
- **`workcenter_group` grouping reuses `filter_cache.get_spec_workcenter_mapping()`.** SPEC→group resolved from the existing cache (`WORK_CENTER_GROUP`); unmapped SPECNAME rows are excluded (PA-06, data-boundary not error). → rejected new hardcoded SPECNAME map: reason — request non-goal; duplicates an existing cache.

## Migration / Rollback
No automated MySQL migration runner exists in this project (confirmed: no `migrations/` dir, no Alembic; the only DDL mechanism is `sync_worker._ensure_mysql_tables()` which runs `CREATE TABLE IF NOT EXISTS` at startup for log/metrics tables only). To avoid coupling feature-table creation to the log-sync worker's lifecycle, the two new tables are created by a standalone idempotent DDL script committed at **`scripts/sql/production_achievement_tables.sql`** (`CREATE TABLE IF NOT EXISTS` for `production_achievement_targets` and `production_achievement_edit_permissions`, matching data-shape §3.26/§3.27 columns + unique constraints). An operator applies it manually against the MySQL OPS database before first deploy — this is deploy-checklist item 5 in ci-gate-contract.md's "production-achievement-kanban Gate Compatibility Note". backend-engineer writes the actual DDL there; do not embed it here. Precondition: `MYSQL_OPS_ENABLED=true` in production (deploy item 4; env-contract.md). Rollback: remove the page's manifest/nav entries (ci-gate rollback items 1); the two tables are left in place (orphaned but harmless, per ci-gate rollback item 3) — no DROP, no Oracle spool cleanup (feature uses no DuckDB spool), no RQ worker to stop.

## Open Risks
- **DDL-drift risk:** because table creation is manual (not enforced by code at startup), a deploy that skips checklist item 5 will surface as 503 on target/permission writes and null denominators on reads — degrades safely (no 500) but silently. Mitigation: the write path's 503 is the operator-visible signal; consider a startup health log if MySQL is enabled but the tables are absent (implementation-planner to scope, non-blocking).
- **`user_identifier` identity match:** `can_edit_targets` compares the whitelist `user_identifier` against the LDAP session username (`session["user"]["username"]`). If admin grants use a different identifier form than the session identity, the gate silently denies. Contract-reviewer/backend-engineer must pin one canonical identity form (recommend session username verbatim).
- **PA-04 three-shift C-band attribution is an UNVERIFIED assumption** (business-rules.md PA-04); three-shift-regime queries must log/flag the unverified rule. Out of scope for acceptance but a correctness caveat for 2020-Q1 historical queries.
- No ADR written: the direct-MySQL vs sync_worker choice is a documented deviation but does not move a persistence substrate, consistency guarantee for existing data, or trust boundary of existing surfaces — it is an additive new path fully captured in Key Decisions above. If a future change tries to route these writes through sync_worker, this design.md is the reversal guard.
