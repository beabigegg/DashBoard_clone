# Design: downtime-analysis-page-redesign

## Summary
Restructure the downtime-analysis page from a three-tab switcher (Charts / Equipment / Events) into a single page: charts on top, a three-tier expandable table below (Tier 1 = Status/Category group UDT/SDT/EGT, Tier 2 = Machine within status, Tier 3 = lazy-loaded Events/JOB per machine). Two ECharts components gain click cross-filter wiring: `BigCategoryChart` sector click sets `chartFilter.big_category`; `DailyTrendChart` legend click sets `chartFilter.status_types`. Both are forwarded as additive optional query params to the existing `equipment-detail` / `event-detail` endpoints, which already serve all views from the in-memory parquet spool via `apply_view()`. No new Oracle query, no spool-namespace change, no rowcount chunking — the change stays fully within ADR-0002 (independent `downtime_analysis_*` namespace) and ADR-0003 (whole-dataset chunking, post-merge reductions untouched). Filtering is applied as a pandas `.isin()` narrow on the already-assembled `events_df` before the existing `_build_equipment_detail` / `_build_event_detail` reducers run.

## Key Decisions

- **DQ-1 — Filter on the backend (Option B).** Pass `big_category` / `status_types` / `resource_id` to the endpoints and re-fetch. Rejected client-side in-memory (Option A): equipment rows are paginated and the three-tier layout needs every machine per status group; client-side filtering only works if all rows are loaded upfront, which makes pagination state and cross-filter narrowing brittle. The backend already reads the full spool per call, so a server filter is a cheap `.isin()` and keeps one source of truth.
- **DQ-2 — Load all equipment rows in one page (Option A).** Request `page_size=200` (the existing hard cap) for the equipment list; ~142 total rows fit a single page, so frontend pagination of the machine list is removed. Tier 1 status grouping and Tier 2 machine nesting are computed client-side from this single full set. Rejected Option B (paginate within status groups): cross-page status grouping is complex and would split a status group across pages. If row count ever exceeds 200, fall back to status-group-scoped fetches keyed by `status_types` — a future, additive change.
- **DQ-3 — Tier 3 lazy-load with composite cache key.** On machine-row expand, call `GET /event-detail?query_id=&resource_id=<id>&status_types=<group-status>`, scoped to the status group the machine sits in. Cache key `${resource_id}|${status_type}` (a machine appears separately under UDT vs SDT). Invalidate the entire Tier 3 cache on re-query (new query button). On chartFilter change, also clear Tier 3 cache: a chartFilter narrows which Tier 1/2 rows exist, so stale Tier 3 entries could attach to a machine no longer shown or under a now-excluded status — clearing is cheap and avoids a silent mismatch.
- **DQ-4 — CSV serialization for `status_types` (Option A): `?status_types=UDT,SDT`.** Matches the established codebase convention — `_csv_param()` in the route module already parses `workcenter_groups`, `families`, `resource_ids` as CSV. Rejected repeated params (Option B): would require a new `request.args.getlist` path inconsistent with every other multi-value param on these blueprints.
- **DQ-5 — DailyTrendChart cross-filter uses legend-click (Option A).** `@legendselectchanged` toggles `chartFilter.status_types` (UDT/SDT/EGT union). Rejected bar-segment click (Option B): date-scoped filtering needs a new backend date param that is explicitly out of scope (non-goal: no new query/param beyond the three named filters), and `event-detail`/`equipment-detail` have no per-day filter today.
- **DQ-6 — Un-import, do not delete, EventDetail.vue / EquipmentDetail.vue.** Remove the imports/usage from App.vue but leave the files on disk for git-history continuity and one-revert rollback. Rejected deletion: the redesign's three-tier components are new and unproven in production; keeping the prior views on disk makes the rollback a single App.vue revert with no file resurrection. A follow-up cleanup change can delete them once the redesign is stable.

## Migration / Rollback
No data migration: no DB schema change, no spool-namespace change (ADR-0002 namespace and `DOWNTIME_BRIDGE_VERSION` untouched), no parquet schema rewrite (so no `tmp/query_spool/downtime_analysis/` cleanup in the runbook). Backend changes are additive optional params that default to no-op; omitting all three returns the pre-existing unfiltered response byte-for-byte, so existing callers and the 410/cache paths are unaffected. Rollback is a single revert of `App.vue` (and the additive route/service kwargs); `EquipmentDetail.vue` / `EventDetail.vue` remain on disk, so the three-tab layout is restorable without un-deleting files. The additive params carry no 410/500 regression risk on the existing endpoints.

## Open Risks
- **R1 (low):** New Tier 1/2/3 components add authored CSS; any unscoped top-level rule bleeds across portal-shell pages. Mitigation: all rules under `.theme-downtime-analysis`, enforced by `npm run css:check` Rule 6.
- **R2 (low):** `StatusMachineJobTable` may reuse shared-ui `DataTable` (12+ consumers). Any change to DataTable must be additive-only; prefer a standalone component if nesting cannot be expressed without altering its prop/emit surface.
- **R3 (low):** 200-row equipment cap is a silent ceiling; if a wide date range yields >200 machines the list truncates. Pin with a test asserting total machine count ≤ cap for the fixture, and document the status-group-scoped fallback.
- **R4 (low):** Tier 3 key `${resource_id}|${status_type}` must match the status label the chart legend emits (UDT/SDT/EGT) exactly; a casing/whitespace mismatch yields a silent empty Tier 3 (Type-A key-mismatch class). Verify against the route's `success_response` wrapper key `events`.
