# Design: resource-history-cache-fix

## Summary
Resource-history has two coexisting spool-key systems that never connect: the route's primary path stores spool under a **filter-inclusive** key (`execute_primary_query` hashes start/end/granularity + all filter params), while the canonical read path (`try_compute_query_from_canonical_spool`) looks up a **canonical** key (date-range + granularity, no filters). The warmup job calls `ensure_dataset_loaded()`, which checks the canonical key but then delegates to `execute_primary_query`, which writes the filter-inclusive key — so the canonical spool is never populated and the canonical path always returns SPOOL_MISS. This change makes the canonical key the single storage key for the warm dataset (removing `granularity` from it so one parquet serves all day/week/month/year switches and all filter combinations via DuckDB view-time filtering), bumps the schema version to invalidate stale parquet, stops the route from silently swallowing canonical errors, and adds an optional Phase-2 view-result cache so repeat granularity/filter switches skip DuckDB recompute. API response shape and source SQL are unchanged.

## Key Decisions

- **D1 (AC-3): Remove `granularity` from the canonical key.** Chosen. `base_facts.sql` stores daily shift-hours per equipment (HISTORYID + TXNDATE); bucketing happens in DuckDB `_query_trend` via `_granularity_bucket_expr`. One parquet then serves all four granularities. Bump `_CANONICAL_BASE_SCHEMA_VERSION` 1 → 2 to invalidate stale spool. *Rejected: keep granularity in key — multiplies spool count 4× and forces a fresh Oracle query on every day→week toggle; directly contradicts ADR-0001 precedent.* Recorded as ADR-0005.
- **D2 (AC-2): `ensure_dataset_loaded()` writes the canonical key directly.** Chosen Option A — a small internal helper queries Oracle for the unfiltered dataset and registers the base + OEE spool under `make_canonical_base_query_id`/`make_canonical_oee_query_id`. *Rejected Option B (dual-write inside `execute_primary_query`) — couples two key systems in the hot path and adds branch complexity; Option A keeps the warm-write isolated.*
- **D3 (AC-1, AC-4): Canonical path is primary; remove `except Exception: pass`.** Chosen. In `/query`, on canonical HIT use it; on SPOOL_MISS (canonical key absent) fall through to `execute_primary_query`, which now also registers the canonical key on an empty-filter Oracle fetch. A genuine canonical runtime error surfaces instead of being silently swallowed. *Rejected: keep the blanket swallow — it masks key-parity regressions exactly like the original bug.*
- **D4 (AC-8): Keep System A (filter-inclusive key) as fallback; defer deprecation.** Chosen. System A remains the Oracle-fetch path for filtered first-queries and for date ranges outside the warm window. Removing it now widens blast radius on a high-risk core. *Rejected: delete System A in this change — lower-risk to consolidate in a follow-up once canonical path is proven warm in production.*
- **D5 (AC-7): Include Phase-2 view-result cache (additive, low risk).** Chosen, behind a TTL env var. After a canonical HIT, `apply_view` still recomputes all six DuckDB queries; cache the full result dict keyed by `hash(canonical_query_id + granularity + sorted_filter_params)`, TTL 300 s (`RESOURCE_VIEW_CACHE_TTL`, default 300). Cache all six structures atomically or none. *Rejected: per-query caching — risks partial/inconsistent view assembly. Deferrable if scope grows, but it carries no correctness risk.*
- **D6: No conflict with the multi-worker startup lock.** Confirmed. The lock in `resource_history_duckdb_cache.py` guards creation of the on-disk DuckDB file. The canonical warmup writes Parquet + Redis pointers via `store_spooled_df`/`register_spool_file` — a different substrate. The two are independent; document in `docs/cache-strategy.md`. No code change to the lock.
- **D7 (AC-6): Stale-parquet invalidation via schema_version bump + post-deploy cleanup.** Schema bump changes the key hash, so old parquet is simply never looked up (no BinderException — lookup misses, falls to Oracle). Redis pointer TTL expires old entries, but orphaned parquet files persist on disk; add `rm tmp/query_spool/resource_dataset/*.parquet` and `rm tmp/query_spool/resource_oee/*.parquet` to the deploy runbook. ci-cd-gatekeeper records this in `ci-gates.md §Rollback Policy`.

## Migration / Rollback
Forward: bump `_CANONICAL_BASE_SCHEMA_VERSION` to 2 so all canonical keys change; deploy; on post-deploy, run the two `rm` commands above to clear orphaned parquet (Redis pointers expire by TTL but disk files do not self-clean). The first `/query` per date range warms Oracle→canonical spool; the next warmup cycle (`WARMUP_INTERVAL_SECONDS`, default 3600 s) pre-warms the 90-day window. No DB migration, no schema/DDL, no API contract change.

Rollback: revert the code; schema_version returns to its prior value, so reverted readers look up the old key and again ignore any v2 parquet (miss → Oracle), no crash. Re-run the same two `rm` commands after rollback to drop v2 orphans. If Redis is down or parquet is incompatible at any point, both paths fall back to the Oracle query path with no error.

## Open Risks
- View-cache TTL of 300 s means a filter/granularity switch can serve up to 5-minute-stale derived numbers within an already-warm dataset; acceptable for a reporting surface but must be stated in `docs/cache-strategy.md`.
- D4 leaves two key systems live during the transition; a future filter param added to System A but not reflected in the canonical view-time filter would diverge silently — the follow-up deprecation change must close this.
- Empty-filter detection in `execute_primary_query` (D3 canonical co-write) must treat "all flags false + all lists empty" as the canonical case; a mis-classification would write the wrong key. Pin with a key-parity unit test (AC-2).
