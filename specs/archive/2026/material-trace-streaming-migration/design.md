# Design: material-trace-streaming-migration

## Summary
Migrates material-trace's heavy Oracle execution from the `pd.concat(chunks)` + post-hoc `_check_memory_guard()` path (`_execute_batched_query`, OOM risk #3 in query-dataflow-unification §1.4) onto the unified `BaseChunkedDuckDBJob` streaming pipeline: Oracle → Arrow RecordBatch → DuckDB/parquet spool, with no DataFrame ever assembled in the worker heap. The change is gated by feature flag `MATERIAL_TRACE_USE_UNIFIED_JOB` (default off). When off, the legacy `_execute_batched_query` / `execute_to_spool` paths are untouched. The frontend spool/view contract (Redis-cached enriched parquet, `_paginate` page slices, CSV export) and the `material_trace` spool namespace/schema stay identical, so each side is independently rollback-safe.

## Affected Components
| component | file path(s) | nature of change |
|---|---|---|
| material-trace job class | `services/material_trace_duckdb_runtime.py` | new `BaseChunkedDuckDBJob` subclass: `pre_query` (resolve container IDs, build ID batches, compute deterministic spool key), `build_chunk_sql` (per ID-batch SQL), `post_aggregate` (register/dedup merged parquet) |
| service entry switch | `services/material_trace_service.py` | flag branch in `execute_to_spool` / `rq_material_trace_job`; legacy `_execute_batched_query`, `_execute_batched_query_to_parquet`, `_check_memory_guard` retained for flag=off |
| feature flag | `core/feature_flags.py`, `contracts/env/*` | register `MATERIAL_TRACE_USE_UNIFIED_JOB` (bool, default off) |
| chunked job base (reuse) | `core/base_chunked_duckdb_job.py` | no change; consumed via `ChunkStrategy.ID_LIST`, `requires_cross_chunk_reduction=False` |
| concurrency (semantics only) | `core/global_concurrency.py` | no code change; semaphore role re-described (see D3) |
| Arrow reader / spool (reuse) | `core/oracle_arrow_reader.py`, `core/query_spool_store.py` | consumed unchanged |

## Key Decisions

- **D1 — `requires_cross_chunk_reduction = False`.** Evidence: `_execute_batched_query` (material_trace_service.py L191-240) builds chunks by Oracle IN-list batching of resolved IDs (`_IN_BATCH_SIZE=1000`, L216-230), then `pd.concat` (L235) and a `drop_duplicates` on `[CONTAINERID, MATERIALLOTNAME, WORKCENTERNAME, TXNDATE]` (L237-238). Callers `forward_query` (L604/L607) and `reverse_query` (L665) then apply only `_enrich_workcenter_group` (row-level `.map`, L125-135) and `_paginate` (row slice, L335-356). There is **no SUM/COUNT/GROUP BY collapsing rows across batches** — the only cross-batch op is a row-level set-union dedup. Per ADR-0003's classification rule, dedup is a row-uniform operation that does not split a logical aggregate across chunk seams, so ID_LIST parallel chunking is seam-safe. The dedup is preserved as a DuckDB `SELECT DISTINCT`/`row_number` step (or `merge_chunks_to_spool` + distinct) in `post_aggregate`, matching the exact 4-column key. → rejected alternative in R2.
- **D2 — Remove the post-hoc `_check_memory_guard()` on the unified path.** The guard (L175-188, called at L239 after `pd.concat`) fires only after the full frame is already in heap — too late to prevent the concat-time OOM (§1.4 病根: post-hoc guards cannot prevent OOM). The streaming path never assembles a DataFrame; Arrow batches flow chunk-by-chunk into DuckDB, which spills to on-disk job DuckDB / parquet (§2.4), making peak memory independent of result size (§6 acceptance). DuckDB on-disk spill is the structural replacement guard. Note the existing streaming spool path `_execute_batched_query_to_parquet` already documents "`_check_memory_guard()` is NOT called" (L257) — this decision extends that proven property to the unified path.
- **D3 — Re-purpose, not re-wire, the heavy-query semaphore.** `global_concurrency.acquire_heavy_query_slot` (Lua CAS over a Redis sorted set, L78-111, fail-open) is unchanged in code. Its semantics shift from "throttle synchronous slow queries blocking gunicorn" to "cap concurrent RQ heavy jobs hitting Oracle" (§4.2 two-level control). The per-job chunk fan-out is bounded separately by `BaseChunkedDuckDBJob.max_parallel` (default 3). No new env, no CAS change.
- **D4 — 503, no silent sync fallback, when flag=on.** Following the eap-alarm-unified-job-poc precedent (always-async + 503), when `MATERIAL_TRACE_USE_UNIFIED_JOB=on` and RQ/Redis is unavailable, the route returns 503 rather than silently degrading to an in-request pandas query (which would reintroduce the OOM path the flag exists to remove). Flag=off preserves whatever the current route behavior is.

## Rejected Alternatives
- **R1 — Keep `pd.concat` with a *pre-hoc* guard (check projected size before concat).** Rejected: a pre-concat estimate cannot bound the actual concat, which allocates the union of all chunk frames simultaneously; the OOM happens *inside* `pd.concat` itself before any post-check runs. Only never-entering-heap (streaming) eliminates it.
- **R2 — `requires_cross_chunk_reduction = True` unconditionally (safe default).** Rejected: the code evidence in D1 shows no cross-chunk aggregation, only row-level dedup. Forcing True would open a per-job DuckDB raw table and a GROUP BY post-stage for no semantic gain, adding the single-writer serialization cost (§5.1) and job-DuckDB lifecycle (§5.4) where the lighter multi-parquet append path (`requires_cross_chunk_reduction=False`, §2.4) suffices. ADR-0003 requires evidence-based classification, not a blanket conservative default.

## Migration / Rollback Strategy
Default `MATERIAL_TRACE_USE_UNIFIED_JOB=off` → route enqueues the legacy `rq_material_trace_job` → `execute_to_spool` → `_execute_batched_query_to_parquet`. Flip on per-environment after parity verification (new vs legacy spool parquet: identical schema, rowcount, and dedup-key uniqueness for the same `(mode, values, workcenter_groups)`). Spool namespace (`material_trace`) and parquet schema are unchanged, so the frontend `/view` and CSV export need no change and roll back transparently. Rollback runbook: (1) set flag off; (2) `rm` any orphan `{DUCKDB_JOB_DIR}/material_trace/*.duckdb`; the canonical `{QUERY_SPOOL_DIR}/material_trace/*.parquet` is format-compatible across both paths and may be left in place (TTL-managed). If a spool was written by a buggy unified run, delete that specific `query_hash` parquet to force re-query on the legacy path.

## Open Risks
- **Dedup fidelity:** the unified `post_aggregate` DISTINCT must use the exact 4-column key from L238 (`CONTAINERID, MATERIALLOTNAME, WORKCENTERNAME, TXNDATE`); a wider/narrower key silently changes rowcount vs legacy. Pin with a parity test.
- **`WORKCENTER_GROUP` enrichment placement:** legacy sync path enriches post-fetch (`_enrich_workcenter_group`), legacy spool path enriches inline per chunk (L306-312). The unified path must enrich inside the chunk/DuckDB stage so spool parquet columns match the existing schema — verify the column set, not just rowcount (cache-spool false-"UNCHANGED" learning).
- **CHAR/Decimal coercion:** legacy parquet path casts `Decimal→float` (L302) and the schema is fixed from the first non-empty chunk (L315-320); the Arrow reader path must reproduce the same dtype coercion or DuckDB cast errors / dtype drift appear at parity check.
- **Spool key determinism:** `make_route_query_hash` (L800-816) must remain the spool key under the unified path so existing spool-hit idempotency (L844-858) and frontend `query_id` reuse keep working.
