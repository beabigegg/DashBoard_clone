# Design: production-achievement-overhaul

## Summary
The 生產達成率 report is restructured from a free-form date-range query into a **2×2 view model** — `DailyView` (當日/前日: single-day, rows = PACKAGE_LF group, columns = D產出/N產出/合計/計畫/達成率) and `CumulativeView` (當月/自訂區間: multi-day, rows = PACKAGE_LF group, columns = 累計計畫/產出/差異/達成率). Both modes share one wire protocol and differ only in how the date window is chosen and whether an hourly warm-cache pre-populates the spool. This **extends, not reverses, ADR-0016**: the server still ships a coarse spool + inline dimension maps and computes zero rollup on the request path, but the spool grain widens SPECNAME→(SPECNAME, PACKAGE_LF) and the client-side DuckDB-WASM rollup grows from one stage into a **2-stage pipeline** (SPECNAME→raw workcenter_group, then merge-map + package-lf-map joins) plus downstream daily-plan joins. 當日/前日 reuse the **existing shared `spool_warmup_scheduler`** by directly invoking `ProductionAchievementJob` (via a progress-report-suppressing subclass), and a new **standalone `/production-achievement-settings` mini-app** administers the 3 new MySQL config tables. A bundled fetch-completeness fix (D6) closes a systematic N-shift-tail under-count that single-day 前日 would otherwise expose.

## Key Decisions
- **D1/D2 opposite defaults are correct, not an inconsistency to normalize.** `package_lf_map` (D1) is sparse/fallback-to-self via `LEFT JOIN … COALESCE(merged_group, raw_package_lf, '(未分類)')`; `workcenter_merge_map` (D2) is explicit-inclusion/exclude-by-absence via `INNER JOIN`. They encode two different real-world questions. Every raw PACKAGE_LF is legitimate output that must always appear (merging is a rare display-normalization exception → absence must mean "show as itself"); only ~12 of ~27 raw workcenter_groups are real production stations, the other 15 (倉/線邊倉/IST/CP…) must never appear (→ absence must mean "exclude"). D2 mirrors the permission table's default-deny idiom (§3.27); D1 mirrors the target table's reuse-as-null idiom (§3.26). A shared default would either drop legitimate package output or flood the report with 15 junk groups. → rejected "make both the same join kind for consistency": it inverts one table's meaning.
- **Warm-cache reuses `ProductionAchievementJob` via a `progress_report()`-overriding subclass, not a parallel Oracle path.** Reuse is safe because `BaseChunkedDuckDBJob.run()` has no RQ-context dependency — `job_id` is just a string for dir-naming/logging. → rejected a hand-written second Oracle→DuckDB→spool path: it would duplicate ADR-0016's seam-safe chunk/`post_aggregate` correctness into a drift-prone twin. The non-obvious trap forcing the *subclass* (not a bare instantiation): `ProductionAchievementJob.progress_report()` calls `update_job_progress()`, which does an **unconditional Redis `HSET` with no TTL and no existence check**; calling `.run()` directly from the warmup module bypasses `enqueue_query_job` registration, so nothing ever expires that key — one orphaned Redis key would leak per warmup cycle forever. The subclass no-ops `progress_report()` to sever that (PA-14).
- **PACKAGE_LF / workcenter-merge / daily-plan rollup stays client-side (ADR-0016 extended one level).** The server keeps shipping one canonical date-range spool + inline maps and computes nothing on the request path; the widened grain and 2-stage pipeline are additive DuckDB-WASM computations on the same principle. This preserves the date-range-only canonical spool key and makes an admin's mapping edit a free client-side re-filter rather than an Oracle re-fetch. → rejected moving the merge/plan joins server-side: it would force the merged dimensions into the spool key (breaking the canonical-key invariant), reintroduce a server-side parity surface, and require a re-spool on every settings edit. Reversal must flip ADR-0016 to `superseded`.
- **D6 closing-chunk is a fetch-completeness fix, not a rollup-locus change, and is bundled deliberately.** The N-shift tail `[end+1 00:00, end+1 07:30)` is attributed by PA-03 back to `end_date`, but a date-only `:chunk_end_excl` (implicit midnight) never fetched it. `post_aggregate`'s existing `GROUP BY … SUM` was always *able* to fold those rows into the `(end_date,'N',…)` group — they simply never arrived; nothing about the aggregation, its cross-seam safety argument, or the client-vs-server locus moves. It is bundled (not deferred) because single-day 前日 turns a previously-invisible edge-of-range under-count (smoothed across multi-day ranges) into a systematic daily under-count of yesterday's N-shift headline — DailyView cannot be correct without it (PA-15).
- **Settings is a standalone drilldown route, not an admin-dashboard tab.** `/production-achievement-settings` is its own mini-app in `STANDALONE_DRILLDOWN_ROUTES` with no drawer entry, reached only via a 設定 button. It is gated by `can_edit_targets` (a wider whitelist than admin), so nesting it under the admin dashboard would mis-model the trust boundary. It is a *satellite of the report*: the report page stays the single owner of mode/station state and the settings detour is designed not to disturb it — exactly what OD-7 (preserve mode/station across the round-trip) needs; a top-level admin-tab context switch would make the return-to-exact-state contract far more awkward. → rejected admin-dashboard tab.
- **The two DuckDB stages stay separate — dimension-widening order forbids collapsing them.** Stage 1 (`pa_rollup_raw`) resolves SPECNAME→raw workcenter_group (PA-06's existing INNER JOIN) while now carrying PACKAGE_LF through the GROUP BY. Stage 2 (`pa_rollup`) joins `workcenter_merge_map` — **keyed on the resolved raw workcenter_group, not SPECNAME** — and `package_lf_map`. The merge table cannot be joined against the SPECNAME-grain spool at all, because the raw-group column it keys on does not exist until Stage 1 produces it. The intermediate `(raw workcenter_group, PACKAGE_LF)` grain is a meaningful, separately-testable seam that also keeps the D1-LEFT vs D2-INNER opposite-join-kind boundary auditable. → rejected one fused multi-join SELECT: it obscures the grain transition and the join-kind boundary.

## Migration / Rollback
The 3 MySQL tables (`scripts/sql/production_achievement_tables.sql`, additive `CREATE TABLE IF NOT EXISTS`) must be **manually applied before** the backend deploy that reads/writes them — same manual-DDL precondition as the existing 2 PA tables (not wired into `sync_worker`). The `_PA_SPOOL_SCHEMA_VERSION` 1→2 bump changes the canonical spool key, so stale v1 parquets are simply never hit again and age out by TTL; `rm -f tmp/query_spool/production_achievement/*.parquet` is an optional fast-forward documented in ci-gates.md §Rollback. The chart replacement (delete `AchievementChart.vue`, add `PlanAchievementStackedChart.vue`) is a **hard cutover with no feature flag**, matching this feature's established no-dual-path precedent (`PRODUCTION_ACHIEVEMENT_USE_UNIFIED_JOB` has never had a legacy fallback branch). A full rollback reverts the one atomic PR across contracts/SQL/worker/services/routes/frontend and: (a) leaves the 3 new MySQL tables **orphaned/unused** — dropping them is optional cleanup, not required, since nothing else reads them; (b) removes the 2 new `_WARMUP_JOBS` entries (otherwise the scheduler keeps warming a removed cache path); (c) removes the settings page's 7 registry locations — the `vite.config.ts` `INPUT_MAP` removal is mandatory (a dangling entry with deleted source blocks the frontend boot), the other 6 are graceful; and (d) re-runs `cdd-kit openapi export` for both output paths after reverting `api-contract.md`. Warm-cache is fail-safe by construction (PA-14): a total scheduler outage degrades to the ordinary 202-poll path, never a hard failure.

## Open Risks
- **Non-goal (backfilled):** month/range warm-caching is explicitly **out of scope** — only 當日/前日 are pre-warmed; 當月/自訂區間 always take the existing async-spool path. Flag as a follow-up only if month-mode latency later proves inadequate.
- **OD-9/OD-10 accepted silent-empty:** an empty `workcenter_merge_map` (MySQL OPS down) collapses the whole report to an empty table via the D2 INNER JOIN, indistinguishable from a genuinely-empty day — confirmed acceptable (no read-time discriminator), but operationally a config-store outage reads as "no output."
- **Settings→report propagation latency:** a saved mapping/plan edit only reaches DailyView after the next warmup cycle (≤ ~`WARMUP_INTERVAL_SECONDS`); disclosed via an OD-5 save-time note, no spool-refresh-on-save. Confirm the latency is acceptable at QA sign-off.
- **D6 changes historical N-shift numbers** intentionally and only at the closing-chunk seam; the regression fixture must assert the corrected total (not just new-row presence) so the intended-only delta is durable.
- `.cdd/code-map.yml` was not consulted for this write (working tree shows it modified); affected-component ranges were grounded by direct reads of every cited source file.
